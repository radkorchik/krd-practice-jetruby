# Выполненное задание по лекции 11 (Active Record)

## 1) Что такое ORM, в чем основная задача?
ORM (Object-Relational Mapping) - это способ работать с таблицами базы данных через объекты и методы языка программирования.

Основная задача ORM:
- убрать ручное написание большого объема SQL в бизнес-логике;
- связать объекты приложения с таблицами БД;
- упростить CRUD-операции и поддержку кода.

## 2) В чем принцип работы паттерна Active Record?
Принцип Active Record: каждая модель обычно соответствует таблице БД, а объект модели соответствует строке таблицы.

Пример:
- класс `Post` <-> таблица `posts`;
- `Post.new`, `save`, `update`, `destroy` выполняют операции записи;
- `Post.find`, `where`, `joins` выполняют чтение.

То есть логика доступа к данным находится прямо в модели.

## 3) Какие ассоциации можно задать в моделях и как?
Основные типы ассоциаций в Rails:

- `belongs_to` - объект принадлежит другому объекту (в таблице есть внешний ключ).
- `has_one` - у объекта одна связанная запись.
- `has_many` - у объекта много связанных записей.
- `has_many :through` - связь многие-ко-многим через промежуточную модель.
- `has_and_belongs_to_many` - многие-ко-многим через join-таблицу без отдельной модели.

Пример:

```ruby
class Book < ApplicationRecord
  belongs_to :author
end

class Author < ApplicationRecord
  has_many :books
end
```

## 4) Что такое миграция? Зачем нужен schema.rb файл?
Миграция - это версия изменения структуры БД (создание таблиц, добавление/удаление колонок, индексов, внешних ключей и т.д.).

`schema.rb` нужен для:
- хранения актуального состояния структуры БД в одном файле;
- быстрого поднятия структуры командой `db:schema:load`;
- удобной синхронизации схемы между разработчиками и окружениями.

## 5) Сколько уровней валидации есть в обычном приложении? Как написать свою?
Обычно используют 3 уровня:

1. Клиентский (frontend) - для быстрого UX-ответа.
2. Контроллерный - базовая проверка входных параметров.
3. Модельный (основной, обязательный) - защита целостности данных перед записью в БД.

Свою валидацию можно написать:

- методом в модели:

```ruby
class Article < ApplicationRecord
  validate :body_presence

  private

  def body_presence
    return if body.present?
    errors.add(:body, 'must be present')
  end
end
```

- отдельным валидатором (`ActiveModel::Validator`) и подключить через `validates_with`.

## 6) Что такое коллбеки и какой их практический смысл?
Коллбеки - это методы, которые Rails запускает автоматически на этапах жизненного цикла модели.

Примеры: `before_validation`, `before_save`, `after_create`, `before_destroy`, `after_commit`.

Практический смысл:
- автоматизировать повторяющиеся действия;
- нормализовать/подготовить данные перед сохранением;
- запускать побочные действия после изменений (логирование, уведомления и т.д.).

Важно: не перегружать коллбеки сложной бизнес-логикой.

## 7) В чем заключается N+1 проблема? Как ее можно решить?
N+1 - это ситуация, когда сначала выполняется 1 запрос за коллекцией, а потом еще N запросов для связанных данных каждого элемента.

Пример проблемы:
- загружаем 10 книг;
- затем для каждой книги отдельно грузим автора.

Решение:
- использовать предварительную загрузку ассоциаций: `includes`, `preload`, `eager_load`.

Пример:

```ruby
books = Book.includes(:author).limit(10)
books.each { |book| puts book.author.last_name }
```

Так количество запросов резко уменьшается.
