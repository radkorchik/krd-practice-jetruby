ОТЧЕТ ПО ЗАДАНИЮ #2: Архитектура Роутера и Модули в Ruby

1. Как работает класс Router (Маршрутизатор)?
Класс Router выполняет роль диспетчера в нашем приложении. Его главная задача — принять пользовательский ввод (имитацию HTTP-запросов: GET, POST, PUT, DELETE) и связать его с нужным методом контроллера для выполнения логики.

Внутри роутера хранится структура данных — хеш @routes (своеобразная "карта маршрутов"). 
Метод `resources` берет класс контроллера (например, PostsController), создает его экземпляр и заполняет этот хеш связками: 
Ключ (название действия/глагола) => Значение (объект метода контроллера).

Роутер использует механизм получения объекта метода: `method(:index)`, `method(:create)` и т.д. Он не вызывает эти методы сразу, а только сохраняет ссылки на них. Позже, когда пользователь выбирает действие в консоли, роутер находит нужную ссылку в хеше и отложенно запускает метод с помощью `.call()`. 
Это классический паттерн проектирования, который делает код гибким: роутер ничего не знает о том, как сохраняются посты, он просто "дергает за нужные ниточки".

2. Причины использования `extend` для модуля Resource:
В Ruby есть два способа подмешать модуль в класс:
- `include` добавляет методы модуля как методы экземпляра (доступны только у созданного объекта через .new).
- `extend` добавляет методы модуля как методы самого класса (вызываются прямо на имени класса).

В нашем приложении (в классе Router) меню вызывается так:
`PostsController.connection(@routes['posts'])`

Мы обращаемся напрямую к самому классу `PostsController`, а не к его экземпляру. Поскольку метод `connection` описан внутри модуля `Resource`, нам жизненно необходимо использовать именно `extend Resource` внутри контроллера. 
Если бы мы написали `include Resource`, Ruby выдал бы ошибку NoMethodError, так как класс сам по себе не знал бы о существовании метода `connection`.
